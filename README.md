# SuperGenius

> 一个基于飞书多维表格的多智能体虚拟招聘组织。

没有谁天生是全能的天才，但一群各自擅长一件事的人协作起来，可以组成一个"超级天才"——这就是SuperGenius想做的事。

SuperGenius把招聘里每个岗位上做事的人，抽象成一个个数字员工(Agent)。他们通过多维表格OpenAPI读写同一张业务表，从岗位立项一路走到Offer发放，每一步动作都留在表里。

## 终极目标

先回到需求，一个好的招聘系统，到底该做到什么？

作为一个还在不断投简历、不断被面试的学生，我们想要的其实只有三件事：

- **Discovery** — 不漏掉对的人
- **Fairness** — 不偏待任何人
- **Match** — 不错配任何岗位

## 按招聘流程介绍

招聘是需求驱动的。下面按阶段展开，每段说清楚这个阶段在做什么、谁来做、他们之间怎么接力。

---

### 阶段一 · 需求立项与JD生成（岗位需要什么样的人）

解决的问题：要先把岗位讲清楚，缺几个人、什么级别、多少预算、多急，然后才有JD。

涉及角色：招聘经理、JD策划官。

#### 招聘经理Hiring Manager

- **职责**：提出岗位需求，审批JD；后续还要负责仲裁分歧和签Offer
- **输入**：用人部门的招聘诉求
- **输出**：岗位记录（级别、预算上下限、紧急度、JD要点）
- **协作**：把需求丢给JD策划官写JD，拿到草案后审批

#### JD策划官JD Strategist

- **职责**：按需求写JD；后面根据漏斗数据回头改JD
- **输入**：岗位需求、历史投递数据、分析师的JD诊断
- **输出**：JD文本（可能多版本）、JD改版提案
- **协作**：从招聘经理那拿需求→写JD→交回去审批；收到分析师的诊断后做改版

---

### 阶段二 · 候选人进入与初筛

候选人投简历，系统把简历解析成结构化数据并打分；新岗位开放时，旧简历也可能被重新翻出来。

涉及角色：候选人、简历筛选官。

#### 候选人Candidate

- **职责**：看岗位描述决定投不投；后面还要参加面试、反向提问、谈薪、做Offer决策
- **输入**：岗位描述、面试问题、Offer内容
- **输出**：简历、面试回答、反问、Offer响应、谈薪诉求
- **人格配置**：稳重型 / 激进型 / 骑驴找马型 / 被动型
- **异常行为**（用来测系统鲁棒性）：面试中途失联、Offer后反悔、临时加薪、被别家撬走
- **协作**：把简历交给筛选官；和面试官对话；向招聘经理提谈薪

#### 简历筛选官Screener

- **职责**：解析简历、按JD打分、初筛放行或淘汰；有新岗位时反向翻人才池
- **筛选维度**：硬门槛（学历/年限/技能）、相关度、成长轨迹、稳定性、红旗信号
- **输入**：新投递简历、JD、历史简历库
- **输出**：结构化简历记录（含分数、理由、决策：通过/待定/淘汰）、反向唤醒提案
- **一致性自检**：同一份简历多次打分，方差超阈值就标记需要复核
- **反向激活**：新岗位开放时扫历史"被淘汰 / Offer没接"的简历，匹配度高的触发二次邀约
- **协作**：收到候选人简历→通过名单交给三位面试官；淘汰的留在人才池；反向唤醒提案告诉招聘经理

---

### 阶段三 · 多维度评估

三位面试官同时打分，各打各的；如果分歧太大就进入辩论；辩不出来再升级给招聘经理仲裁。

涉及角色：技术面试官、业务面试官、文化面试官、招聘经理（仲裁）。

#### 技术面试官Tech Interviewer

- **职责**：评估候选人的技术硬实力
- **维度**：编码能力、系统设计、技术深度、学习能力、工程习惯
- **输入**：简历、JD、候选人回答
- **输出**：面试题（动态生成）、分维度评分（0-10）、评语（要引用候选人原话）、辩论发言
- **协作**：独立评分→参加会诊→如果自己和另两位差3分以上就进辩论（最多3轮）→还谈不拢就上报招聘经理

#### 业务面试官Business Interviewer

- **职责**：评估业务理解和项目经验的真实度
- **维度**：业务理解、贡献可验证性、优先级判断、行业认知、沟通表达
- **输入 / 输出 / 协作**：结构和技术面试官一样，只是维度不同

#### 文化面试官Culture Interviewer

- **职责**：用STAR框架看价值观和团队适配
- **维度**：价值观匹配、抗压模式、协作风格、自我认知、动机稳定性
- **输入 / 输出 / 协作**：结构同上；评分理由必须基于候选人讲到的具体事件

**三人会诊规则**

1. 三位面试官拿到同一份简历和岗位，各自独立出题、独立打分、独立写评语
2. 打完分进入会诊
3. 如果有人和另两位差3分以上，触发辩论（最多3轮，全程写进辩论日志表）
4. 还不收敛就上报招聘经理仲裁

---

### 阶段四 · Offer审批与响应

招聘经理决定是否发Offer、是否加钱；候选人决定接、谈、拒或者放鸽子。

涉及角色：招聘经理、候选人。

#### 招聘经理Hiring Manager（本阶段）

- **职责**：看面试评估和辩论日志→决定发不发Offer、要不要加码
- **输入**：面试评估表、辩论日志、候选人谈薪诉求
- **输出**：仲裁结论、Offer审批记录（批准 / 加码 / 拒绝）
- **协作**：向候选人发Offer；收到谈薪诉求决定授权加薪

#### 候选人Candidate（本阶段）

- **响应**：接受 / 谈薪 / 比较其他Offer / 拒绝 / 放鸽子
- **协作**：和招聘经理就Offer互动

---

### 阶段五 · 数据分析与闭环优化

整个漏斗的数据汇到分析师手里，变成周报、JD诊断、面试官偏差报告，再反哺回前面几个阶段。

涉及角色：招聘分析师。

#### 招聘分析师Analyst

- **职责**：跑全量数据，产出能落地的判断
- **分析维度**：
  - **漏斗转化率**：投递→初筛→面试→Offer→入职每一级的流失率
  - **时间效率**：响应时长、每环节耗时、岗位平均填坑周期
  - **质量信号**：Offer接受率、面试到场率
  - **面试官校准**：每位面试官的平均分、方差、辩论触发率、被仲裁改判率
  - **JD健康度**：投递量、初筛通过率、候选人画像偏离度
- **输入**：全量业务表数据
- **输出**：周报（飞书消息推送）、JD诊断、面试官偏差报告、异常预警
- **协作**：
  - 周报→招聘经理（考虑要不要调整岗位要求或补充渠道）
  - JD诊断→JD策划官（改版JD）
  - 面试官偏差→招聘经理（校准面试官）
  - 人才池信号→简历筛选官（触发反向激活）
